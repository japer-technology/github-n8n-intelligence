# How This Will Work — At the Coal Face

> A practical walkthrough of what actually happens when a user interacts with this Githubified n8n system. No theory — just the sequence of events, who does what, and where things land.

---

## The Setup (One-Time, Already Done)

Before any user opens an issue, these things are already in place:

| Component | Where it lives | What it does |
|-----------|---------------|--------------|
| **Agent workflow** | `.github/workflows/github-n8n-intelligence-agent.yml` | Listens for issues and comments, runs the AI agent |
| **n8n executor workflow** | `.github/workflows/n8n-execute.yml` | Installs n8n and executes workflow JSON files |
| **Template fetcher workflow** | `.github/workflows/fetch-n8n-templates.yml` | Pulls 400+ community templates weekly from n8n.io |
| **AI agent code** | `.github-n8n-intelligence/lifecycle/agent.ts` | The brain — reads issues, calls the LLM, posts replies |
| **Agent skills** | `.github-n8n-intelligence/.pi/skills/` | Teaches the AI how to build and execute n8n workflows |
| **Workflow library** | `workflows/templates/` (400+ JSON files) + `workflows/utilities/` + `workflows/integrations/` | Pre-built n8n workflows the agent can reference or execute |
| **Credential templates** | `credentials/*.template.json` | Show users which GitHub Secrets to set up |
| **LLM API key** | GitHub Secrets (e.g. `OPENAI_API_KEY`) | Powers the AI agent |
| **n8n encryption key** | GitHub Secrets (`N8N_ENCRYPTION_KEY`) | Lets n8n decrypt credentials at runtime |

## Scenario 1: User Opens a New Issue (The Main Flow)

This is the primary interaction pattern. A user opens an issue on the repository and the AI agent responds.

### Step 1 — User Opens the Issue

The user clicks **New Issue** on the GitHub repository. Today there are no issue templates — the user writes a free-form title and body. For example:

> **Title:** Build me a workflow that labels new issues based on their title
>
> **Body:** When a new issue is opened, check the title. If it contains "bug", add the `bug` label. If it contains "feature", add `enhancement`. Otherwise add `triage`. Post a comment confirming the label was applied.

The user clicks **Submit new issue**.

### Step 2 — GitHub Fires the Webhook

GitHub emits an `issues.opened` event. This event matches the trigger in `github-n8n-intelligence-agent.yml`:

```yaml
on:
  issues:
    types: [opened]
```

A new GitHub Actions run starts automatically. No human intervention needed.

### Step 3 — Authorization Check

The **run-agent** job begins. Its first step checks: *does this user have write access (or higher) to the repository?*

- **If yes** → A 🚀 rocket reaction is added to the issue. Processing continues.
- **If no** → A 👎 reaction is added. The job stops. This prevents random users on public repos from burning LLM credits.

The user sees the 🚀 appear on their issue within seconds — visual confirmation that the agent is working.

### Step 4 — Repository Checkout & Environment Setup

The Actions runner:
1. Clones the full repository (with complete git history, for session context)
2. Checks that `.github-n8n-intelligence/` exists (safety gate)
3. Installs Bun (fast TypeScript runtime)
4. Restores cached `node_modules` if the lockfile hasn't changed
5. Runs `bun install --frozen-lockfile`

This takes roughly 10–20 seconds, mostly cached.

### Step 5 — The Agent Reads the Issue

`agent.ts` runs. Here's what it does, in order:

1. **Extracts the issue content** from the GitHub webhook payload — the title and body the user wrote.
2. **Checks for a prior session**: looks for `.github-n8n-intelligence/state/issues/<issue-number>.json`. For a new issue, this file doesn't exist yet — fresh conversation.
3. **Skips reserved prefixes**: if the issue body starts with a backtick, `~`, `@`, etc., the agent exits (those are for other agents or raw code blocks).
4. **Validates the LLM API key** is present in the environment.

### Step 6 — The Agent Calls the LLM

The agent invokes the pi coding agent with the issue content as the prompt:

```
pi --mode json --provider openai --model gpt-5.4 \
   --session-dir .github-n8n-intelligence/state/sessions \
   -p "Build me a workflow that labels new issues based on their title\n\nWhen a new issue is opened..."
```

The LLM has access to the agent's skills — including:

- **n8n-workflow-builder** — knows the JSON structure of n8n workflows, all common node types, the GitHub node's 50+ operations, how connections work, and how credentials map to GitHub Secrets.
- **n8n-executor** — knows how to trigger `n8n-execute.yml`, where logs go, what environment variables are needed, and how to troubleshoot failures.
- **issue-triage** — can categorise and prioritise the request.

The LLM generates a response. For our example, it would likely:
1. Build a complete n8n workflow JSON file with nodes for issue triage
2. Explain what each node does
3. Tell the user how to execute it

### Step 7 — The Agent Posts a Reply

The agent extracts the LLM's response from the streamed JSONL output (capped at 60,000 characters — GitHub's comment limit) and posts it as a comment on the issue.

The user sees something like:

> **🤖 Agent response:**
>
> I've created a workflow that labels issues based on their title. Here's the workflow JSON saved to `workflows/integrations/issue-label-by-title.json`:
>
> ```json
> {
>   "name": "Label Issues by Title",
>   "nodes": [
>     { "name": "Start", "type": "n8n-nodes-base.start", ... },
>     { "name": "Triage Logic", "type": "n8n-nodes-base.function", ... }
>   ],
>   "connections": { ... }
> }
> ```
>
> To execute this workflow, add the `n8n-execute` label to this issue or run:
> ```
> gh workflow run n8n-execute.yml -f workflow_file=workflows/integrations/issue-label-by-title.json
> ```

### Step 8 — Session State is Saved

The agent:
1. Creates `.github-n8n-intelligence/state/issues/<issue-number>.json` — maps this issue to a session file
2. Saves the full conversation transcript to `.github-n8n-intelligence/state/sessions/<timestamp>.jsonl`
3. Commits everything: `git add -A && git commit -m "minimum-intelligence: work on issue #42"`
4. Pushes to the repo (with a retry loop — up to 10 attempts with exponential backoff, rebasing on conflicts)

This is critical: **the conversation is now in Git**. If the user comments again, the agent can pick up where it left off.

### Step 9 — Reactions Updated

- The 🚀 rocket reaction remains (signals "agent processed this")
- A 👍 is added on success (or 👎 if something went wrong)

### Step 10 — GitHub Pages Deploys

The **run-gitpages** job runs after the agent finishes. It deploys the `public-fabric/` directory to GitHub Pages. This is automatic — no user action needed.

## Scenario 2: User Comments on an Existing Issue (Continuing the Conversation)

The same flow fires, but with a key difference at Step 5:

1. GitHub emits `issue_comment.created`
2. The workflow checks: is the commenter a bot? If yes, skip (prevents infinite loops of the agent replying to itself)
3. `agent.ts` finds the existing session file at `state/issues/<issue-number>.json`
4. It loads the prior conversation transcript from `state/sessions/<timestamp>.jsonl`
5. The LLM receives the full conversation history plus the new comment
6. The agent responds with context — it remembers what was discussed before

**This is how multi-turn conversations work.** Each issue is a persistent conversation thread. The session state in Git is the agent's memory.

## Scenario 3: Executing an n8n Workflow

Once the agent has built a workflow JSON file and committed it, there are three ways to actually run it:

### A. Manual Dispatch (User Triggers It)

```bash
gh workflow run n8n-execute.yml -f workflow_file=workflows/integrations/issue-label-by-title.json
```

Or via the GitHub Actions UI: **Actions → n8n Workflow Executor → Run workflow → enter the file path**.

### B. Issue Label Trigger

The `n8n-execute.yml` workflow also triggers on `issues: [opened, labeled]`. Adding a specific label to an issue can trigger execution.

### C. Scheduled (Cron)

```yaml
schedule:
  - cron: '0 */6 * * *'  # Every 6 hours
```

Scheduled workflows run automatically. The default runs `workflows/utilities/hello-world.json`.

### What Happens During Execution

1. Actions runner installs n8n globally: `npm install -g n8n`
2. Configures SQLite database and encryption key from secrets
3. Imports the workflow: `n8n import:workflow --input=<file>`
4. Executes headlessly: `n8n execute --id=1`
5. Captures output to `execution-output.log`
6. Commits the log to `executions/latest/<timestamp>.log`
7. If triggered by an issue: posts the execution result as a comment

## Where Does Issue Templating Fit In?

Currently, there are **no issue templates** in `.github/ISSUE_TEMPLATE/`. Users write free-form issues and the AI agent handles whatever they throw at it.

Issue templates would improve this by giving users structured starting points. Here's how they'd work:

### How a Templated Issue Would Work

A user clicks **New Issue** and sees a template picker (e.g. "🔧 Build a Workflow", "▶️ Run a Workflow", "🐛 Report a Bug"). They choose one and get a pre-filled form:

```markdown
### What do you want to automate?
<!-- Describe the automation in plain language -->

### Trigger
<!-- What should start this workflow? (e.g. new issue, push, schedule, manual) -->

### Actions
<!-- What should happen? (e.g. send Slack message, add label, create file) -->

### Required services
<!-- Which external services are involved? (e.g. GitHub, Slack, email) -->
```

The user fills in the sections and submits. From there, the exact same flow described in Scenario 1 fires — the agent reads the structured content and produces a better response because the input is more organized.

### Why Templates Help

| Without templates | With templates |
|---|---|
| User writes "make something that does the thing" | User fills in trigger, action, and services |
| Agent has to guess intent | Agent gets structured input |
| Higher chance of back-and-forth clarification | First response is more likely to be right |
| New users don't know what to ask for | Templates show what's possible |

### Template Categories That Would Make Sense

| Template | Purpose | Key fields |
|----------|---------|------------|
| **🔧 Build a Workflow** | Ask the agent to create an n8n workflow from a description | Trigger, actions, services, schedule |
| **▶️ Execute a Workflow** | Run an existing workflow from the library | Workflow file path, input data |
| **📋 Browse Templates** | Search the 400+ community template catalogue | Search terms, category filter |
| **🐛 Report a Problem** | Report a bug with a workflow or the agent | Workflow file, error message, steps to reproduce |
| **💬 General Question** | Ask the agent anything about n8n or the repo | Free-form |

## The Four GitHub Primitives (Why This Works Without a Server)

Everything maps to four built-in GitHub features:

| Primitive | Replaces | How |
|-----------|----------|-----|
| **GitHub Actions** | n8n server, execution engine | Ephemeral VMs run `n8n execute` on demand — no always-on server |
| **Git repository** | n8n database (SQLite/Postgres) | Workflow JSON, session state, execution logs — all version-controlled |
| **GitHub Issues** | n8n visual editor (Vue.js UI) | Users describe automations in natural language; AI builds the workflows |
| **GitHub Secrets** | n8n credential store (AES-256) | API keys, tokens, encryption keys — injected at runtime |

No external servers. No databases. No Docker. No infrastructure to manage. The entire automation platform runs on GitHub's existing primitives.

## What Could Go Wrong (And What Happens)

| Failure | What the user sees | What happens internally |
|---------|-------------------|----------------------|
| User lacks write access | 👎 reaction on issue | Auth step fails, job stops |
| LLM API key not set | 👎 reaction, error in Actions log | Agent validates key, exits with error |
| LLM returns empty response | 👎 reaction | Agent detects empty output, reports failure |
| Git push conflict (parallel agents) | Nothing visible — agent retries | Up to 10 push attempts with rebase, exponential backoff |
| `.github-n8n-intelligence/` missing | Notice in Actions log, no reply | Safety check skips gracefully |
| n8n workflow execution fails | ❌ in execution comment | Error captured in log, posted to issue |

## Summary

1. **User opens an issue** (free-form or from a template)
2. **GitHub Actions fires** within seconds
3. **AI agent reads the issue**, loads conversation history, calls the LLM
4. **Agent replies** with a workflow, explanation, or answer — posted as a comment
5. **Everything is saved to Git** — conversation state, workflow files, execution logs
6. **User continues** by commenting on the issue; agent picks up the conversation
7. **Workflows execute** via `n8n execute` on Actions runners when triggered
8. **No servers, no infrastructure** — just GitHub
